Introduction

About This Guide

You're a Sales Engineer. You're not a developer. You weren't hired to write code. This guide is going to make you dangerous anyway.

This guide was built from 30 days of building real tools after hours — usually three nights a week, occasionally full weekends when I had the house to myself. Limited formal dev background, but eager to acquire more knowledge. The learning curve was steep for about a week. Then it accelerated faster than anything I'd experienced in this career.

The goal was simple: learn as much as I could about building with AI to improve how I work and live, then share those learnings with other SEs. Not to give you a toolkit — to show you what I built, what I learned, and the possibilities. So you can see what's out there and skip the mistakes I made.

This guide covers the complete stack — from first installation to working automated lab environments — with practical steps you can start immediately.

Rule of thumb: if you've explained your project to Claude more than twice in the same week, you forgot to write a CLAUDE.md.

Chapter 3

Key Concepts — The Glossary You'll Actually Use

You're going to hear these terms constantly. Each one explained in plain English.

TermPlain English Explanation
Markdown (.md files) Simple text formatting that every tool in this stack understands. # for headers, ** for bold, - for lists. It's the common language between you, Claude, GitHub, and Notion. Takes 5 minutes to learn.
CLAUDE.md A markdown file you put in your project folder. Claude Code reads it automatically at the start of every session. It's a briefing document: "here's what this project is, here's what we've built, here's what to do next." This single file is the difference between Claude starting confused and Claude picking up exactly where you left off. The most important concept in this guide.
project.md A companion to CLAUDE.md with more detail: infrastructure specifics, API endpoints, deployment instructions, known issues. CLAUDE.md is the executive summary. project.md is the technical reference.
MCP (Model Context Protocol) A standard that lets Claude talk to external tools directly. Instead of copy-pasting between Claude and Notion, MCP lets Claude read and write to Notion on its own. Same for n8n, databases, file systems, and dozens of other tools. Configure once, forget forever.
Skills Packaged instructions that teach Claude how to do specific things well. Like handing a new hire a procedures manual. Claude Code comes with built-in skills, and you can create custom ones. They live in ~/.claude/skills/.
System Prompt Instructions you give Claude before a conversation starts. Defines its role, rules, and context. Think of it as a job description for AI.

Chapter 4

The Plan-Then-Execute Pattern

This is the single most important workflow in the guide. Everything else builds on it.

Claude Code has two modes that work together:

Plan Mode — Claude reads your project, analyzes your request, and produces a detailed plan: files to create, changes to make, commands to run. It executes nothing. You review, question, adjust.

Bypass Permissions — Once you've approved the plan and you're confident Claude understands the task, you enable auto-accept. Claude executes the entire plan without pausing at each step. It creates, runs, tests, iterates — all autonomously.

The Pattern

  1. Describe what you want to build
  2. Turn on Plan Mode — Claude maps out the full approach
  3. Review: Does it understand the architecture? Is it touching files it shouldn't? Is the approach correct?
  4. Adjust if needed ("don't use that library" or "start with the backend first")
  5. Approve. Turn on Bypass. Let it build.
  6. Come back, review output, test, iterate.

Without Plan Mode, you're approving each action one at a time — like reviewing a construction project one brick at a time instead of looking at the blueprint. With it, you see the whole plan, approve once, and let the builder work.

The Rules

Always use Plan Mode when:

  • Starting a new project
  • Making architectural changes
  • Working with unfamiliar libraries or frameworks
  • Modifying critical infrastructure or security-related code
  • You're not 100% confident Claude understands what you want

Only use Bypass Mode when:

  • You've reviewed the plan and it's correct
  • The task is straightforward and you trust Claude's approach
  • You're iterating on something you've already built
  • Time matters and you're willing to review output afterward

Never use Bypass Mode when:

  • Claude hasn't produced a plan first
  • You're working in production
  • The task involves deleting files or data
  • You don't understand what Claude is about to do
Review the blueprint. Then let the builder work.

Chapter 5

Dos and Don'ts — Especially for a Cybersecurity Company

You work at Sophos. Your team will immediately (and correctly) ask: "What data is being sent where?" This section answers that.

DO

  • Use CLAUDE.md in every project. It's the single biggest productivity multiplier. No CLAUDE.md = starting from zero every session.
  • Document as you build. Update Notion and project files after every session. Future you will be grateful.
  • "Describe, Build, Test, Iterate." Don't try to spec everything perfectly upfront. Tell Claude what you want, see what it builds, adjust.
  • Keep secrets in .env files. Add .env to your .gitignore. Commit a .env.example with placeholder values.
  • Ask Claude to explain. "What does this code do?" makes you learn faster than copying blindly.
  • Use Claude to write your documentation. It's genuinely better at structured docs than most humans. Let it.

DON'T

  • Never paste customer data into any AI tool. No PII. No customer environment details. No case data. No exceptions.
  • Never paste credentials into a prompt. No API keys, passwords, tokens, or secrets. If Claude needs an API key, it goes in a .env file.
  • Never commit .env files to GitHub. This is how secrets leak. It happens to professionals constantly. Don't be that commit.
  • Never trust AI output blindly. Review generated code before running it, especially anything touching infrastructure or customer-facing systems.
  • Never use AI-generated code in production without review. It's a starting point, not a finished product. Test it. Read it. Understand it.

Chapter 6

Claude Chat vs Claude Code

Plan in Chat. Build in Code.

One of the most common early mistakes. Takes 60 seconds to fix.

Claude Desktop (the chat app) is where you think, plan, write specs, brainstorm, review outputs, and draft non-code content. It's your strategist.

Claude Code (inside VS Code or terminal) is where you build. It creates files, writes code, runs commands, connects to tools via MCP, and executes. It's your builder.

Same intelligence. Different capabilities.

Chapter 7

The Memory Pattern

CLAUDE.md and Persistent Context

The quality of what AI builds for you is directly proportional to the context you give it. This is not a suggestion. It's physics.

What Goes in CLAUDE.md

This file lives in your project root. Claude Code reads it automatically. Every time. No setup required.

# CLAUDE.md — [Project Name]

## Overview
[2-3 sentences: what this project is and what it does]

## Infrastructure
[What's running, where, how to access it]

## Completed Work
[What's been built so far — bullet points]

## Pending Tasks
[What's next — in priority order]

## Key Decisions
[Choices made and why — so Claude doesn't re-debate them]

## File Paths
[Important directories and files]

## Credentials
[Where to find them — NEVER the actual values]

The Memory Loop

This is the pattern that makes everything work:

  1. Start a session — Claude reads CLAUDE.md, knows exactly where things stand
  2. Build stuff — Claude creates, modifies, tests
  3. End the session — Ask Claude: "Update CLAUDE.md with what we accomplished and what's next"
  4. Next session — Claude reads the updated CLAUDE.md, picks up seamlessly

Getting Started

Your Path Forward

You don't need a perfect plan. You need to start building. Here's how.

This guide has shown you the complete stack — tools, concepts, patterns, real projects. Now it's time to apply it. You don't need to master everything at once. Pick a starting point, build something small, then expand from there.

First Steps (Start Here)

Week 1: Install and Explore

  1. Set up your tools — VS Code, Claude Desktop, Claude Code, GitHub account
  2. Build your first thing — Ask Claude Code to create a simple HTML dashboard or automation script
  3. Create your first CLAUDE.md file — Start the memory habit from Day 1
  4. Push to GitHub — Get comfortable with git add, git commit, git push

Week 2: Add Context and Connections

  1. Set up Notion — Create a project workspace, document what you're building
  2. Connect MCP tools — Link Claude to Notion, enable web search, add Gamma
  3. Install Docker — Run your first container (n8n is a good starting point)

Topics to Explore (Pick What Interests You)

Once you have the basics down, choose topics based on what you want to build. You don't need to do all of these — pick the ones that solve real problems for you.

Beginner Level

  • Automation workflows — Use n8n to automate reports, notifications, data collection
  • Documentation tools — Build a knowledge base, automate meeting notes, create onboarding guides
  • Data dashboards — Visualize SE metrics, customer data, lab inventory (Sanitize sensitive data)

The Core Pattern (Repeat This Forever)

Every project you build from here follows the same workflow:

  1. Plan — Describe what you want in Claude Chat, refine the approach
  2. Create CLAUDE.md — Set up the memory file before you start building
  3. Build — Use Claude Code with Plan Mode → review → Bypass
  4. Document — Update Notion, update CLAUDE.md, commit to GitHub
  5. Iterate — Test, improve, add features

This pattern works whether you're building a one-hour automation or a month-long infrastructure project. The tools change. The pattern doesn't.

What Success Looks Like

After your first week:

  • You've built something that works (even if it's simple)
  • You have a CLAUDE.md habit
  • Your projects are on GitHub
  • You understand Plan-Then-Execute

After your first month:

  • You've automated at least one repetitive task
  • You're comfortable with Docker and n8n
  • You've connected Claude to your other tools via MCP
  • You've shown a colleague something you built

After three months:

  • You're building tools people at Sophos actually use
  • You're teaching others the patterns from this guide
  • You're thinking "I could build that" when you see gaps in your workflow
  • The SE team is asking you how you built things so fast

Final Advice

Start small. Your first project doesn't need to be LabOps. Build a script that saves you 10 minutes a week. Build a dashboard that answers one question you ask repeatedly. Build a tool that makes one part of your job slightly less annoying.

Document everything. Future you will thank past you. CLAUDE.md isn't optional. GitHub commits aren't optional. Notion documentation isn't optional. These aren't overhead — they're the difference between building once and rebuilding every session.

Share what you learn. The best way to solidify your skills is to teach someone else. Walk a teammate through this guide. Show them a tool you built. Answer their questions. You'll find gaps in your own understanding — that's how you get better.

The tools will keep improving. Claude gets smarter. New MCP connectors appear. Docker images get easier. But the patterns in this guide — Plan-Then-Execute, CLAUDE.md, document as you build — those don't change. Learn the patterns. The rest is just applying them to new problems.

Go build something.
